## 実施した要件対応

### 1. 支出申請ページの申請タイプ選択から「事前申請の精算」を削除

対応内容:

- `src/index.html` の申請フォームから `精算` 用 option を削除
- `src/javascript.html` 側で、支出申請モードと精算モードを別状態として管理
- 精算は「精算する」ボタンからのみ開始する方式に変更

結果:

- 通常の支出申請画面では `事後` と `事前` のみ選択可能
- `精算` は UI 上で申請タイプとして選べない

### 2. 「証憑(レシート)」を「領収書(レシート)」に変更

対応内容:

- ユーザー画面の文言を `領収書(レシート)` に統一
- 管理者画面の一覧見出しも `領収書` に変更
- 共通文言は `src/accounting-domain.ts` の `RECEIPT_LABEL` に集約

### 3. 添付ファイルを PDF のみに制限

対応内容:

- クライアント側:
  - `src/index.html` の `input[type=file]` の `accept` を `application/pdf` に変更
  - `src/javascript.html` で `.pdf` または `application/pdf` 以外を弾く
- サーバー側:
  - `src/expenditure-policy.ts` の `validateAttachment()` で最終検証
  - `事後` / `精算` は PDF 必須
  - `事前` は添付不要

結果:

- フロントでもバックでも PDF 以外は拒否
- 将来フロントを迂回してもサーバー側で防げる

### 4. 精算手続きページの申請タイプ欄を削除

対応内容:

- `src/index.html` に申請タイプ欄コンテナ `typeSelectGroup` を残しつつ
- `src/javascript.html` の精算モードで `display:none` に制御

結果:

- 精算開始時は申請タイプ欄が見えない
- 精算は `requestType = '精算'` を内部的に固定して送信

### 5. 精算手続きページの文言変更

対応内容:

- 精算時の添付欄ラベルも `領収書(レシート)` に統一

### 6. 事前請求がある状態で追加の事前請求を禁止

対応内容:

- `src/expenditure-policy.ts` に業務ルールを移設
- 同一団体の `事前` レコードについて、以下のいずれかのステータスが存在する場合に新規 `事前` を禁止:
  - `申請中`
  - `承認済`
  - `未精算`

制御箇所:

- サーバー: `ExpenditureRequestPolicy.assertCanCreateRequest()`
- 画面: `formBootstrap.requestAvailability` を使って選択肢を disabled 化

### 7. 事後請求がある状態で追加の事後請求を禁止

対応内容:

- 同一団体の `事後` レコードに `申請中` がある場合のみ新規 `事後` を禁止
- `承認済`, `却下`, `精算完了` などはブロック条件に含めない

制御箇所:

- サーバー: `ExpenditureRequestPolicy.assertCanCreateRequest()`
- 画面: `requestAvailability` をもとに `事後` option を disabled 化

## データ仕様と業務ルールの整理

### T_Expenditure の appendRow 仕様

`processForm()` で追加する列順は以下の通り。

1. 申請 ID
2. 登録日時
3. 団体 ID
4. タイプ
5. ステータス
6. 合計金額
7. 内容文字列
8. ファイル URL
9. 固定値 `未`

今回の初期ステータス:

- `事前` の新規申請: `未精算`
- `事後` の新規申請: `申請中`
- `精算` の新規申請: `申請中`

注意:

- 現行仕様上、精算も `T_Expenditure` に別レコード追加する運用のまま維持している
- 既存列 9 の意味は今回深掘りせず、既存仕様に合わせて `未` を維持

### 申請制限ロジック

#### 事前請求

同一 `groupId` 内に以下のいずれかがある場合、新規 `事前` を禁止:

- `type === '事前' && status === '申請中'`
- `type === '事前' && status === '承認済'`
- `type === '事前' && status === '未精算'`

#### 事後請求

同一 `groupId` 内に以下がある場合、新規 `事後` を禁止:

- `type === '事後' && status === '申請中'`

#### 精算開始条件

同一 `groupId` 内に以下がある場合のみ精算可能:

- `type === '事前' && status === '未精算'`

### 残高計算

残高計算で利用済として加算するステータス:

- `承認済`
- `精算完了`

### 添付ルール

- `事前`: 添付不要
- `事後`: PDF 必須
- `精算`: PDF 必須

## 管理者検索仕様

### 検索条件オブジェクト

`getAdminDashboardData(criteriaInput)` へ渡している条件は概ね以下。

```ts
{
  idQuery: string,
  dateFrom: string | null,
  dateTo: string | null,
  groups: string[],
  types: string[],
  amountMode: 'none' | 'exact' | 'min' | 'max' | 'range',
  amountValue: number | null,
  amountMin: number | null,
  amountMax: number | null,
  contentQuery: string,
  sortBy: 'date' | 'id',
  sortOrder: 'asc' | 'desc'
}
```

### 金額検索の挙動

- `none`: 金額条件なし
- `exact`: 完全一致
- `min`: 指定額以上
- `max`: 指定額以下
- `range`: `amountMin` 以上かつ `amountMax` 以下

### 文字列検索の挙動

- ID: 部分一致
- 内容: 部分一致
- 正規化は `trim().toLowerCase()` ベース

### 並び替え

- `sortBy = date`
  - 主キー: 日付
  - 副キー: ID
- `sortBy = id`
  - 主キー: ID
  - 副キー: 日付

## 次の作業者向けの注意点

### 1. `src/test.js` は既に削除状態だった

`git status` 確認時点で `src/test.js` は削除済みだった。  
今回こちらでは復元していない。  
この削除がユーザー作業か過去作業かは未確認なので、必要なら履歴確認を行うこと。

### 2. `精算` の扱いは既存仕様を踏襲している

今回の修正では UI 上は精算手続き専用画面に寄せたが、保存先は従来通り `T_Expenditure` のまま。  
事前請求レコード自体を更新して精算完了にする設計には変えていない。  
もし将来「精算時は元の事前申請レコードのステータスも自動変更する」などにしたい場合は、今回の対象外。

### 3. 管理者画面の検索候補は現存データから動的生成

- 団体名候補: `T_Expenditure` の `groupName`
- タイプ候補: `T_Expenditure` の `type`

そのため、候補はデータに存在する値のみ表示される。

### 4. import に `.ts` 拡張子を付与している

理由:

- Node の `--experimental-strip-types` でテスト実行するため

この形でも今回の `npm run build` は成功している。  
将来 build 周りを変更する場合は、拡張子解決ルールに注意すること。

### 5. フロントだけでなくサーバーでも制約をかけている

今回の制約は以下の 2 段構え。

- UI 側で選べない / 送れない
- `processForm()` 側でも再検証

今後 UI を改修する場合でも、サーバー側の業務ルールを外さないこと。

## 10. `mise` によるツール管理への集約

### 目的

- ローカルと CI で Node / コマンド実行経路をそろえる
- 手元環境依存を減らす
- `clasp` を必要時だけ使う構成にする

### 対応内容

追加 / 更新した主なファイル:

- `mise.toml`
- `package.json`
- `.github/workflows/deploy.yml`
- `.gitignore`

実施内容:

- `mise.toml` を作成し、Node を `22.14.0` に固定
- `install`, `ci-install`, `build`, `test`, `push`, `deploy`, `watch` を `mise` task として定義
- `clasp` は常駐 tool ではなく `push` / `deploy` / `watch` の task-specific tool として定義
- `package.json` の `build` / `test` / `push` / `watch` は `mise run ...` を呼ぶ構成に変更
- GitHub Actions も `mise run ci-install` / `mise run build` / `mise run deploy` を使うよう変更

### 確認コマンド

```sh
mise tasks ls
mise run build
mise run test
```

### 補足

サンドボックス環境では `MISE_STATE_DIR=/tmp/...` を付けて実行した。  
通常環境では不要。

## 11. GitHub Actions / `clasp` デプロイ不具合の修正

### 発生した問題

GitHub Actions で順に以下のエラーが発生した。

1. `jdxcode/mise-action` が存在せず action 解決失敗
2. `Project settings not found.`
3. `Project contents must include a manifest file named appsscript.`
4. `doGet` が見つからない
5. `Unexpected token ; line: ... file: code.gs`

### 修正内容

#### 11.1 `mise-action` 修正

- workflow の action を `jdx/mise-action@v3` に修正

#### 11.2 `.clasp.json` 不足対応

- workflow 内で `.clasp.json` を生成する step を追加
- `rootDir` を `./dist` とする構成に統一

#### 11.3 `appsscript.json` を build 対象へ追加

対応ファイル:

- `src/appsscript.json`
- `build.js`

実施内容:

- `src/appsscript.json` を正規ソースとして追加
- `build.js` で `src/*.html` だけでなく `src/appsscript.json` も `dist/` へコピーするよう修正

#### 11.4 GAS エントリポイントの公開

対応ファイル:

- `src/code.ts`
- `test/gas-entrypoints.test.ts`

実施内容:

- bundle 後に tree-shaking で落ちないよう、`doGet`, `processForm`, `processInquiry` などを `Object.assign(globalThis, {...})` で明示公開
- build 後の `dist/code.js` を検査するテストを追加

#### 11.5 GAS 互換ターゲットへ修正

対応ファイル:

- `build.js`

実施内容:

- esbuild の `target` を `es2019` に下げ、GAS が解釈できない構文が残らないよう修正

### 再現手順

```sh
mise run build
mise run deploy
```

GitHub Actions では `.clasp.json` と `~/.clasprc.json` が揃っていることを確認すること。

## 12. 管理画面の軽量化

### 背景

管理ダッシュボードや問い合わせ一覧で「読み込み中が長い」「重い」という問題があった。

### 対応ファイル

- `src/accounting-spreadsheet-repository.ts`
- `src/code.ts`
- `src/admin.html`
- `src/admin_script.html`
- `test/accounting-spreadsheet-repository.test.ts`
- `test/admin-script-architecture.test.ts`
- `test/admin-bootstrap-template.test.ts`

### 実施内容

#### 12.1 サーバー側

- `M_Admin`, `M_Users`, `M_ItemMaster` を `CacheService` でキャッシュ
- `getDataRange()` 乱用をやめ、`getLastRow()` と一括 `getValues()` に寄せた
- 管理画面の初期表示で不要なユーザー向け集計をしないようにした

#### 12.2 クライアント側

- 管理画面の初回表示に必要な支出一覧とフィルタ候補をテンプレートへ埋め込み
- 問い合わせ一覧はタブを開いたときだけ遅延読込
- 検索と並び替えはブラウザ側データで実施
- ステータス更新後は全件再取得せずローカル状態を更新

#### 12.3 表示改善

- 初期表示はフィルタなし全件表示
- 団体名検索 / タイプ検索候補も初期表示時点で利用可能

### 確認方法

1. 管理画面を開く
2. 初回表示で全件が出ることを確認
3. 団体名 / タイプ候補が即座に出ることを確認
4. 問い合わせタブは開いたタイミングでだけ読み込まれることを確認

## 19. 今後の保守で特に気をつける点

- 会計タイプを変更するときは旧データの互換正規化も忘れない
- UI だけでなく `ExpenditureRequestPolicy` のサーバー制御も必ず更新する
- 管理画面の重さ対策として、サーバー再取得よりクライアント側処理を優先する
- `build.js` の静的ファイルコピー対象を外すと CI / deploy が壊れる
- GAS エントリポイントは `globalThis` 公開を維持する
- `docs/test-data` に実データ由来の fixture があるため、改修テストの材料として再利用しやすい
